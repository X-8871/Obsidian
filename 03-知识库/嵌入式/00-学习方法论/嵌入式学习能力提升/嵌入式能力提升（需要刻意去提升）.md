**1. 抽象能力：练“先画层，再看码”**
每学一个东西，先不要问“代码怎么写”，先画三层：

```text
它解决什么问题？
它对上层提供什么能力？
它依赖下层什么条件？
```

例如 UART：

```text
问题：两个设备异步传字节
对上：提供 send/receive byte 或 stream
对下：依赖时钟、GPIO复用、电平、波特率、采样、缓冲
```

练法：每学一个外设、协议、模块，都写一张“问题-抽象-依赖”卡片。  
目标：你看到任何技术，第一反应不是 API，而是它在系统里的位置。

**2. 系统思维：练“数据流 + 状态流 + 错误流”**
嵌入式系统不要只看函数调用，要看三条流：

```text
数据怎么流？
状态怎么变？
错误怎么传播？
```

比如一个传感器报警系统：

```text
数据流：传感器 -> 滤波 -> 判断 -> 状态机 -> 显示/报警/日志
状态流：NORMAL -> WARN -> ALARM -> RECOVER
错误流：传感器失败 -> 超时 -> 降级 -> 记录 -> 提示
```

练法：每做一个项目，都画这三条线。  
如果你画不出来，说明你还没掌握系统。

**3. Debug 直觉：练“证据优先”**
Debug 直觉不是玄学，是大量“现象-证据-假设-验证”的积累。

每次遇到 bug，不要马上改代码。先写：

```text
现象是什么？
预期是什么？
我看到的证据是什么？
可能原因有哪些？
哪个最可能？为什么？
怎么用最小实验验证？
```

例如串口没输出：

```text
可能是波特率错、TX引脚错、串口没初始化、printf没重定向、程序没跑到那里、电源/下载问题。
```

然后一个个排，而不是凭感觉改。

练法：建立自己的 `DEBUG_LOG.md`，每次问题都记录：
- 现象
- 最后原因
- 错误判断
- 正确证据
- 下次遇到类似问题先查什么

三个月后你的 debug 直觉会明显变强。

**4. 架构能力：练“边界感”**
嵌入式架构不是上来搞复杂分层，而是知道什么应该分开，什么不该耦合。

你每写一个模块都问：

```text
这个模块负责什么？
它不应该负责什么？
它暴露什么接口？
它隐藏什么细节？
如果硬件换了，哪里需要改？
如果需求变了，哪里需要改？
```

例如：
- `BSP_Button` 只负责读按键电平
- `InputController` 负责消抖、长按、短按语义
- `AppController` 负责状态变化
- `AlertController` 负责报警策略

如果一个文件里既读 GPIO，又判断老人风险，又控制蜂鸣器，又发网络请求，这就是边界坏了。

练法：每周找一个模块，写一段“模块职责说明”。  
目标：能说清楚“这个模块为什么存在”。

**5. 批判 AI 的能力：练“审查而不是接受”**
你要把 AI 当实习生，不是当老师。它写完代码后，你问它：

```text
这段代码依赖哪些假设？
哪些地方可能错？
有没有改变原有行为？
有没有过度设计？
有没有隐藏状态？
失败时会怎样？
怎么验证？
```

然后你自己也检查一遍。

练法：让 AI 每次输出后附带：

```text
Assumptions:
Risks:
Verification:
Simpler alternative:
```

你不要只看答案，要看它有没有把风险说清楚。

**6. 实践能力：做“闭环项目”，不是碎片练习**
点灯、串口、I2C 都是碎片。真正提升来自闭环项目。

一个好嵌入式练习项目要包含：

```text
输入：传感器/按键/串口命令
处理：滤波/判断/状态机
输出：LED/OLED/蜂鸣器/串口日志
异常：超时/断线/错误码
验证：测试步骤和日志
```

你可以设计一个最小训练项目：

```text
环境监测报警器
- AHT20 读温湿度
- OLED 显示
- 按键切换阈值
- 超阈值蜂鸣器报警
- 串口输出状态
- 传感器断线进入 ERROR
- 写 VERIFY_GUIDE
```

这比刷 20 个外设例程更有效。


**最重要的判断标准**
你有没有提升，不看你看了多少教程，而看这 5 件事：

```text
你能不能解释一个模块为什么存在？
你能不能画出数据流、状态流、错误流？
你能不能在改代码前列出证据？
你能不能指出 AI 代码的风险？
你能不能设计验证方法证明它工作？
```

如果这 5 个能做到，你已经不是“会用开发板”，而是在形成真正的嵌入式工程能力。

**7.实际生活中该怎么做**

每次学习一个器件/功能时，要在[[问题-抽象-依赖卡片]]记录卡片，并以该器件/功能为中心建立知识图谱
每次学习一个模块时，要在[[模块职责说明]]写一份说明并以该模块为中心建立知识图谱
每次完成构建一个最小系统时，要在[[数据流_状态流_错误流_三流解释]]解释三流是什么
每次报错时在[[现象-证据-假设-验证链条]]写出报错原因，建立debug直觉

Debug能力和AI审查能力以后再想办法解决，先着手解决以上三个知识网络的能力
然后每周坚持做闭环项目，从小的开始做起
