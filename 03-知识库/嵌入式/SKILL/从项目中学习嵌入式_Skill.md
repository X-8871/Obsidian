# Skill：从真实项目中学习嵌入式知识

> 本 Skill 提炼自一次完整的"从项目出发的嵌入式知识特训"实践。
> 适用于：有真实嵌入式项目代码，但缺乏系统理论基础的学习者。

---

## 一、这个学习方法适合什么人

- 手上有一个真实跑起来的嵌入式工程，但不清楚底层在做什么
- 不想啃枯燥的教材，希望带着"这是干什么用的"去学
- 需要形成一份可以长期复用的个人知识库

---

## 二、学习的核心流程（四个阶段）

```
阶段一：读代码，整理大纲
    AI 遍历项目源码
    → 提炼出可学的知识模块（不要把学习目标定在"理解业务"，而是"提炼出通用技术原理"）

阶段二：分模块出题
    每个模块出 10~15 道专业考题
    → 题目必须和实际代码强相关（源码出现过的函数、宏、配置）
    → 插入源代码片段，让学习者知道"这段代码背后是什么原理"

阶段三：学习者作答，AI 严格批阅
    → 批阅必须严谨，不可为了鼓励而附和错误答案
    → 对于盲点题，给出详尽的"扫盲讲解"而非一笔带过
    → 对于答对的题，简短确认即可，不需要重复解释

阶段四：整理知识库（不按问题列举，而是按框架归纳）
    → 把相关知识点耦合成专题框架
    → 剔除项目特定业务细节，只保留通用技术原理
    → 形成"原理 → 机制 → 陷阱 → 对策"的闭环结构
```

---

## 三、模块划分参考（以嵌入式项目为例）

| 模块 | 核心主题 |
|------|----------|
| 模块一：硬件接口 | GPIO / ADC / I2C / UART / 总线冲突 |
| 模块二：RTOS 系统 | Tick / Stack / Heap / 调度优先级 / 看门狗 |
| 模块三：软件架构 | 状态机 / 分层解耦 / 纯函数 / 防误报 |
| 模块四：网络通信 | Wi-Fi / BLE / HTTP / HTTPS / 云端协议 |
| 模块五：调试排错 | Backtrace / Watchdog / 日志系统 / 条件编译 |

> **出题策略**：重要且易错的模块（如内存管理、状态机、RTOS 调度）可多出几题；
> 结构性强的模块（如网络协议、调试工具）可适当减少题量。

---

## 四、出题的质量要求

1. **题目要和真实代码对应**：不出抽象概念题，要出"这行代码这样写，背后是什么原因"的题目
2. **不出冷门偏题**：考的是"这个项目中工程师必须掌握的"，不是考试大纲题
3. **难度递进**：在每个模块中，先考基础感知（中断/轮询），再考机制细节（ISR 禁止阻塞），最后考系统级设计（Super Loop 架构）
4. **允许插入源码**：出题时在题目旁边贴上相关代码片段，学习者看到真实代码才有画面感

---

## 五、批阅的原则（AI 行为规范）

- **严禁附和**：学习者说错了，必须直接指出，不能为了"鼓励"而说"你的方向是对的"
- **盲点先扫**：学习者回答"不清楚/不知道"时，给出完整的、带类比的深度讲解
- **答对简洁确认**："满分"或"完全正确"+ 一句补充细节即可，不重复解释
- **有结论有证据**：任何纠正都要给出技术依据（函数名、代码行、原理图）
- **不吝惜深度**：学习者追问"详细解释"时，必须给出真实工程师水平的详解

---

## 六、知识库整理的格式规范

整理知识库时，绝对**不能**按问题流水账罗列（第1题...第2题...）。
必须按"技术亲缘关系"重组为专题框架：

### ✅ 正确结构（专题耦合式）

```markdown
## 专题：RTOS 调度模型与并发安全

### Tick 与任务状态机
### 看门狗 (WDT)：死机的终极保底
### 大循环套小循环（Super Loop）并发模型
### GPIO 中断的上下半部（Top/Bottom Half）
```

### ❌ 错误结构（题目罗列式）

```markdown
**第11题：Tick 的相对性**
**第12题：vTaskDelay vs while**
**第13题：pdMS_TO_TICKS 宏**
```

**专题内部的标准写法**：每个知识点遵循"原理 → 机制 → 陷阱 → 对策"的闭环。

---

## 七、学习者脾性适配清单

> 本条目基于实际学习过程中的行为偏好总结，供 AI 辅导时参考：

| 特征 | 应对方式 |
|------|----------|
| **不接受模糊附和** | 严格、直接、有据可依地批阅 |
| **遇到盲点希望深度扫盲** | 用类比（如厨房、炸弹）配合原理讲解 |
| **重视知识库的完整性** | 每轮学完必须及时更新，不能有遗漏 |
| **不需要业务细节** | 剔除项目特定描述，只保留通用技术原理 |
| **知识库要有结构而非罗列** | 按框架耦合，而非按题目编号列举 |
| **希望插入源码** | 出题和讲解时，贴上项目中实际使用的代码片段 |
| **对内容精简度敏感** | 已掌握的知识点只需一句话收录，不需要反复解释 |

## 八、可复用的考题模板

```
【题目类型：原理追问】
「代码中这样写：[代码片段]，为什么不能用更简单的写法 [简单写法]？」

【题目类型：陷阱识别】
「如果有人在 [位置/函数] 里写了 [错误操作]，会发生什么后果？」

【题目类型：机制类比】
「用一句话解释 [技术概念] 是什么？（不使用技术术语，用生活比喻）」

【题目类型：设计决策】
「为什么我们选择 [方案A] 而不是 [方案B]？从 [资源/可靠性/扩展性] 角度分析。」
```

---

## 九、知识库的持续维护规则

- 每个模块学完**当轮次就更新**，不要积攒到最后统一整理（容易遗漏）
- 更新后检查：是否遗漏了学习者单独追问的深度问题？
- 每次更新前先遍历整个对话窗口，对照现有文档逐一确认覆盖情况
- 当知识库变大时，按"专题耦合"而非"按模块分组"进行重组
