# WiFiManager 状态机

## 一、什么是状态机

状态机是一种编程思维，把系统当前所处阶段抽象成几个稳定状态，再规定什么条件会让它切换。

在 WiFiManager 中：

- **状态** = 设备在联网流程中处于哪个阶段
- **事件** = 触发状态切换的外部或内部信号
- **状态切换** = 收到事件后，根据成功/失败决定下一个状态

### 为什么联网必须用状态机

Wi-Fi 配网和联网不是"写几行代码顺序执行完就结束"，而是：

1. 先启动一个动作
2. 系统在后台处理
3. 完成后通过事件通知结果
4. 根据结果决定下一步

整个过程是 **异步事件驱动** 的，无法用简单的 if-else 覆盖所有情况。

## 二、状态定义

WiFiManager 定义了 **7 个平级状态**，没有层级关系，设备一次只能处于其中一个状态。

| 状态枚举值 | 含义 | 说明 |
|-----------|------|------|
| `IDLE` | 空闲 | 初始状态，还没开始任何流程 |
| `PROVISIONING` | 配网中 | 正在 BLE 广播，等待手机发送 Wi-Fi 凭据 |
| `CONNECTING` | 连接中 | 已有 Wi-Fi 凭据，正在尝试连接路由器 |
| `CONNECTED` | 已联网 | 成功获取 IP，设备具备网络通信能力 |
| `RECONNECTING` | 重连中 | 之前已连上过，但掉线了，正在自动重连 |
| `DISCONNECTED` | 已断开 | 与路由器断开，且不在重连流程中 |
| `FAILED` | 失败 | 初始化或流程中发生不可恢复的错误 |

```c
// WiFiManager.h
typedef enum {
    WIFI_MANAGER_STATE_IDLE = 0,
    WIFI_MANAGER_STATE_PROVISIONING,
    WIFI_MANAGER_STATE_CONNECTING,
    WIFI_MANAGER_STATE_CONNECTED,
    WIFI_MANAGER_STATE_RECONNECTING,
    WIFI_MANAGER_STATE_DISCONNECTED,
    WIFI_MANAGER_STATE_FAILED,
} wifi_manager_state_t;
```

## 三、事件分类

状态切换由 **3 类核心事件** 驱动，每一类下面有多个具体事件 ID。

### 3.1 NETWORK_PROV_EVENT（配网事件）

来自 BLE 配网流程，是"用户是否成功把 Wi-Fi 凭据发给设备"的信号。

| 事件 ID | 含义 | 状态切换 |
|---------|------|---------|
| `NETWORK_PROV_START` | BLE 配网开始广播 | → `PROVISIONING` |
| `NETWORK_PROV_WIFI_CRED_RECV` | 收到手机发来的 SSID/密码 | 不改变状态（仅记录日志） |
| `NETWORK_PROV_WIFI_CRED_SUCCESS` | 凭据验证通过 | → `CONNECTING` |
| `NETWORK_PROV_WIFI_CRED_FAIL` | 凭据无效（认证错或 AP 找不到） | → `PROVISIONING`（退回） |
| `NETWORK_PROV_END` | 配网结束，清理资源 | 不改变状态 |

### 3.2 WIFI_EVENT（Wi-Fi 事件）

来自 Wi-Fi 驱动层，是"设备和路由器之间的连接状态"的信号。

| 事件 ID | 含义 | 状态切换 |
|---------|------|---------|
| `WIFI_EVENT_STA_START` | Wi-Fi STA 模式已启动 | → `CONNECTING`（并发起连接） |
| `WIFI_EVENT_STA_DISCONNECTED` | 与路由器断开连接 | → `RECONNECTING` 或 `PROVISIONING` |

### 3.3 IP_EVENT（IP 事件）

来自网络协议栈，是"设备是否真正具备网络通信能力"的信号。

| 事件 ID | 含义 | 状态切换 | 代码中是否处理 |
|---------|------|---------|--------------|
| `IP_EVENT_STA_GOT_IP` | DHCP 成功分配 IP 地址 | **→ `CONNECTED`（唯一入口）** | 是 |
| `IP_EVENT_STA_LOST_IP` | IP 地址丢失 | 不处理 | 否（断线丢 IP 由 `STA_DISCONNECTED` 隐式处理） |

> 注意：WiFiManager 注册事件时指定了 `IP_EVENT_STA_GOT_IP`（不是 `ESP_EVENT_ANY_ID`），所以 `LOST_IP` 不会被回调接收。丢失 IP 的场景实际由 `WIFI_EVENT_STA_DISCONNECTED` 中的 `s_ip_string[0] = '\0'` 覆盖。

### 状态流转对照表

| 当前状态 | 收到事件 | 下一状态 |
|---------|---------|---------|
| `IDLE` | Init 初始化失败 | `FAILED` |
| `IDLE` | provisioned=true | `CONNECTING`（走已配网路径） |
| `IDLE` | provisioned=false | `PROVISIONING`（走 BLE 配网） |
| `PROVISIONING` | `NETWORK_PROV_CRED_SUCCESS` | `CONNECTING` |
| `PROVISIONING` | `NETWORK_PROV_CRED_FAIL` | `PROVISIONING`（退回） |
| `CONNECTING` | `IP_EVENT_STA_GOT_IP` | `CONNECTED` |
| `CONNECTING` | `STA_DISCONNECTED` | `RECONNECTING` |
| `CONNECTED` | `STA_DISCONNECTED` | `RECONNECTING` |
| `RECONNECTING` | 重连成功 + 拿到 IP | `CONNECTED` |
| `RECONNECTING` | 持续重连失败 | `FAILED` |

## 五、核心机制

### 5.1 状态存储

```c
// 唯一的全局状态变量
static wifi_manager_state_t s_state = WIFI_MANAGER_STATE_IDLE;
```

### 5.2 状态切换函数

```c
static void set_state(wifi_manager_state_t state)
{
    if (s_state == state) return;   // 相同状态不做任何事
    s_state = state;                // 更新状态
    ESP_LOGI(TAG, "wifi_state=%s", WiFiManager_GetStatusString());
}
```

- **整个文件只有 `set_state()` 能修改 `s_state`**
- 切换时自动打印当前状态，方便调试
- 重复切换相同状态会被忽略

### 5.3 事件分发入口

所有事件统一由 `wifi_event_handler()` 接收处理：

```c
static void wifi_event_handler(void *arg, esp_event_base_t event_base,
                               int32_t event_id, void *event_data)
{
    if (event_base == NETWORK_PROV_EVENT) { ... }
    if (event_base == WIFI_EVENT) { ... }
    if (event_base == IP_EVENT) { ... }
}
```

### 5.4 关键结论

- **事件是输入，状态是输出**：收到什么事件决定状态怎么变
- **`CONNECTED` 只有唯一入口**：`IP_EVENT_STA_GOT_IP`
- **`CONNECTING` 与 `RECONNECTING` 的区别**：前者是首次连接，后者是掉线后恢复
- **`PROVISIONING` 与 `RECONNECTING` 的区别**：前者还在等用户发凭据，后者已有凭据只是暂时断线

## 六、状态查询接口

WiFiManager 提供以下查询函数，供上层（如 AppController）使用：

| 函数 | 返回值 | 用途 |
|------|--------|------|
| `WiFiManager_GetState()` | `wifi_manager_state_t` | 获取当前原始状态值 |
| `WiFiManager_IsConnected()` | `bool` | 是否已联网（最常用） |
| `WiFiManager_GetStatusString()` | `const char*` | 状态全称（日志用） |
| `WiFiManager_GetStatusShortString()` | `const char*` | 状态简称（OLED 显示用） |
| `WiFiManager_GetIpString()` | `const char*` | 当前 IP 地址 |

```c
// 最常用的查询——判断设备是否已联网
bool WiFiManager_IsConnected(void)
{
    return s_state == WIFI_MANAGER_STATE_CONNECTED;
}
```

## 七、状态机思维要点

看状态机代码时，不要一上来盯细节，先问三个问题：

1. **当前有哪些状态？** → 7 个：IDLE / PROVISIONING / CONNECTING / CONNECTED / RECONNECTING / DISCONNECTED / FAILED
2. **每个状态是什么意思？** → 见第二节表格
3. **什么事件会让它切换？** → 见第三节事件分类

用这组图来看你的工程：

```
PROVISIONING： 先有鸡（配网）还是先有蛋（凭据）
CONNECTING：  正在打电话（拨号中）
CONNECTED：   电话接通了（能说话了）
RECONNECTING：信号断了自动重拨
FAILED：      电话线被剪了（彻底凉了）
```

## 八、完整流程一句话总结

> 配网把凭据送进 NVS → 驱动拿凭据去连路由器 → DHCP 给设备发 IP → `CONNECTED` 状态告诉全世界"我上线了"

事件驱动本质：**先启动动作，结果靠事件回来通知，事件成功与否决定状态怎么变。**
