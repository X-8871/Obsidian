# 1. 配网到底在解决什么问题

配网解决的核心问题只有一个：

- **让设备知道它应该连哪个 Wi‑Fi，以及密码是多少**

配网的本质流程是：

1. 设备先和手机或其他工具建立一个临时通信通道
2. 用户把 Wi‑Fi 名称(SSID) 和 密码 告诉设备
3. 设备把这些信息保存起来
4. 设备再用这些信息去连接路由器
5. 连接成功并拿到 IP（设备自己被分配的IP）后，才算真正联网

所以要分清三件事：

- **配网**：把 Wi‑Fi 凭据配给设备
- **连 Wi‑Fi**：设备用凭据去连路由器
- **联网成功**：设备拿到 IP，能正常访问网络

也就是说：

- 配网不是上网本身
- 配网是"为上网做准备"

---

# 2. Wi‑Fi 基础概念

## 1. SSID

就是 Wi‑Fi 的名字。

## 2. Password

就是这个 Wi‑Fi 的密码。

## 3. 路由器（手机也可以开热点）

路由器的作用是：

- 提供 Wi‑Fi 热点
- 给设备分配 IP
- 帮设备转发到互联网

所以在家庭场景里，嵌入式设备一般不是直接连云，而是：

- 先连路由器
- 再通过路由器出去

## 4. STA 模式

STA = Station

意思是：

- 设备作为"客户端"去连接别人的 Wi‑Fi

比如：

- 手机连家里路由器
- ESP32 连家里路由器

这都属于 STA 模式。

## 5. AP 模式

AP = Access Point

意思是：

- 设备自己开热点，让别人来连它

比如：

- 设备发出一个热点叫 DEVICE_SETUP_xxx
- 手机连这个热点

这就是 AP 模式。

## 6. IP 地址

设备连上 Wi‑Fi 后，还需要拿到一个 IP 地址。
你可以把 IP 理解成：

- 设备在网络里的门牌号

没有 IP：

- 设备虽然可能已经连上了 Wi‑Fi
- 但还不能正常发 HTTP / MQTT 请求

所以真正联网成功的标志不是"Wi‑Fi 连上了"，而是：

- **拿到 IP**

## 7. DHCP

DHCP 是一种自动分配 IP 的机制。

意思是：

- 设备连上路由器后
- 路由器自动给它分配一个 IP

所以通常流程不是：

- 连上 Wi‑Fi 就结束

而是：

1. 连上路由器
2. DHCP 分配 IP
3. 拿到 IP
4. 才能正常联网

---

# 3. 配网方式有哪些

## 1. BLE 配网

通过低功耗蓝牙把 Wi‑Fi 凭据发给设备。

流程大致是：

- 设备开启 BLE 配网服务
- 手机连上设备的 BLE
- 手机把 SSID/密码 发给设备
- 设备保存后再去连 Wi‑Fi

优点：

- 用户体验通常比较好，手机也可以开热点

缺点：

- 需要 BLE 支持
- 手机端通常需要 App 或专门工具

## 2. SoftAP 配网

设备自己开一个 Wi‑Fi 热点，让手机连它。

流程大致是：

- 设备开启热点
- 手机连设备热点
- 手机打开网页或 App
- 输入家里 Wi‑Fi 名和密码
- 设备收到后保存并切回 STA 模式去连路由器

优点：

- 不一定需要 BLE
- 用户可以直接用浏览器

缺点：

- 用户要先切 Wi‑Fi 到设备热点
- 体验有时没 BLE 流畅
- 要写网页代码

## 3. 串口配网

通过串口工具把 Wi‑Fi 信息写进去。

## 4. 固件写死 Wi‑Fi

把 Wi‑Fi 名和密码直接写在代码里。

## 5. 扫码 / NFC / 其他辅助方式

有些产品会用：

- 二维码
- NFC
- 手机系统共享网络凭据

# 4. BLE 配网基础

先抓住一句话：

- BLE 配网 是让手机通过蓝牙把 Wi‑Fi 名称和密码 发给设备

所以 BLE 在这里是：

- **临时配置通道**

## 1. 为什么用 BLE 配网

因为设备第一次上电时没有 Wi‑Fi 凭据，没法直接上网。
这时需要一个"先和手机说上话"的办法。

## 2. BLE 配网过程里要理解的几个概念

### 设备名 / service name

- 给人看的名字
- 比如 PROV_123ABC
- 手机扫描 BLE 时能看到它

### UUID

- 给程序看的"服务标识"
- 告诉手机：这个 BLE 服务是什么类型
- 这里是配网服务的协议标识

### PoP

- Proof of Possession
- 可以理解成配网口令或校验凭证
- 用来确保不是谁都能随便给设备发 Wi‑Fi 信息

你这个项目里就有：

- service name
- UUID
- PoP

## 3. BLE 配网的大致流程

1. 设备启动 BLE 配网服务
2. 手机扫描到设备
3. 手机连接这个 BLE 服务
4. 手机通过配网协议发送：
    - SSID
    - password
5. 设备收到凭据
6. 设备保存到 NVS
7. 设备退出 BLE 配网
8. 设备切到 Wi‑Fi STA
9. 设备连接路由器并获取 IP

所以 BLE 配网只负责前半段：

- **把 Wi‑Fi 凭据送进设备**

## 4. 你在工程里至少要会看懂什么

对于 BLE 配网，嵌入式工程师至少要能看懂这些：

- 配网服务什么时候启动
- 设备广播的名字是什么
- UUID 在哪里设置
- PoP 在哪里配置
- 手机发送凭据后，设备在哪个事件里收到
- 配网成功后，设备什么时候开始连 Wi‑Fi

# 5. NVS / 持久化存储

NVS 是 **非易失性存储**。

一句话理解：

- 断电重启后数据不会丢的存储区

在配网流程里，NVS 非常关键，因为设备不能每次开机都重新问用户要 Wi‑Fi 密码。

所以 BLE 配网拿到凭据后，必须保存下来。

## 1. NVS 在配网里解决什么问题

它解决的是：

- **设备如何记住已经配过的 Wi‑Fi 防止断电重新配**

## 2. 在配网逻辑里，NVS 一般承担什么数据

最常见就是：

- SSID
- password
- 配网完成标记
- 有时还会存设备参数、token、用户配置等

你这个项目最核心的是：

- 是否已经配过网
- 如果配过，保存的 Wi‑Fi 凭据是什么

## 3. 为什么 WiFiManager 一开始要初始化 NVS

因为它需要先能访问"历史保存的联网信息"。

所以会先做：

- nvs_flash_init()

之后才能判断：

- 设备以前有没有保存过 Wi‑Fi 凭据

也就是：

- 是否需要重新进入配网流程

## 4. 清除配网信息是什么意思

清除配网信息，本质上就是：

- 把 NVS 里保存的 Wi‑Fi 配置删掉

删掉之后设备下次启动就会表现得像"从没配过网"，重新走 BLE 配网。

所以你项目里那个：

- 长按 8 秒重置配网

本质上就是：

- 清 NVS 里的 Wi‑Fi 配置
- 然后重启

## 5. 你要掌握到什么程度

对 NVS，这里你至少要会：

- 它是干嘛的
- 它为什么对配网是必须的
- 它和"是否已配网"的关系
- 清掉它为什么能重新配网

# 6. 网络接口和 Wi‑Fi 驱动基础

这一层是把"已经拿到凭据"变成"真正联网"的关键。

一句话理解：

- BLE 把凭据送进设备
- netif + Wi‑Fi 驱动 负责拿这些凭据去真正连路由器

## 1. 什么是网络接口

网络接口就是：

- 设备在网络里的"接入口对象"

在这个项目里，最重要的是：

- Wi‑Fi STA 网络接口

意思是：

- 设备以客户端身份连接路由器

你可以把它想成：

- 软件里的"无线网卡对象"

## 2. 为什么要先初始化 esp_netif

因为后面要创建和管理网络接口。

esp_netif_init() 为后面的

- 创建 Wi‑Fi STA 接口
- 获取 IP
- 绑定协议栈

提供稳定底座。

所以它负责的是：

- **网络接口框架初始化**

## 3. 什么是 Wi‑Fi 驱动

Wi‑Fi 驱动就是：

- 真正控制 ESP32 Wi‑Fi 硬件工作的那层

比如这些动作都归它管：

- 设置 STA 模式
- 启动 Wi‑Fi
- 发起连接
- 接收断线事件

所以这句：

- esp_wifi_init(...)

不是联网成功，而是：

- 把 Wi‑Fi 这套硬件/驱动子系统准备好

## 4. esp_wifi_start() 和 esp_wifi_connect() 的区别

这两个很容易混。

### esp_wifi_start()

- 启动 Wi‑Fi 子系统
- 相当于把无线功能打开

### esp_wifi_connect()

- 发起连接动作
- 用当前保存的 SSID/密码去连路由器

你可以理解成：

- start = 开机
- connect = 去拨号连接

## 5. 为什么创建 Wi‑Fi STA netif 这么重要

因为设备最终要作为：

- Station
- 去连接现有路由器

而不是：

- 自己永远当热点（AP）

所以必须有一个对应的网络接口对象。

在 ESP-IDF 里就是：

- esp_netif_create_default_wifi_sta()

它相当于：

- 创建了设备的"客户端无线网卡"

## 6. Wi‑Fi 驱动和网络接口怎么配合

这两个可以这样分工：

- esp_wifi
    - 管 Wi‑Fi 硬件和连接动作
- esp_netif
    - 管网络接口、协议栈接入、IP 相关能力

配合起来才形成真正的联网链路：

1. 建立 STA netif
2. 初始化 Wi‑Fi 驱动
3. 启动 Wi‑Fi
4. 发起连接
5. 获取 IP
6. 联网成功

## 7. 对你现在最重要的理解

你先不要把它们学成一堆 API，要先建立这个脑图：

- BLE
    - 把 Wi‑Fi 凭据送进设备
- NVS
    - 把这些凭据保存下来
- netif
    - 提供网络接口框架
- Wi‑Fi 驱动
    - 负责真正连接路由器
- IP
    - 表示设备真正进入网络

所以链路其实是：

- **BLE 配网 -> NVS 保存 -> Wi‑Fi 驱动连接 -> netif 拿到 IP**

# 7. 事件驱动模型

这是你理解 WiFiManager 的关键。
一句话先记住：

- Wi‑Fi 配网和联网不是"写几行代码顺序执行完就结束"
- 它是"你先启动流程，后面系统不断通过事件通知你发生了什么"

所以它是 **事件驱动** 的。

## 1. 什么叫事件驱动

事件驱动的意思是：

- 某件事情不是你同步等出来的
- 而是系统在后台发生后，主动通知你

比如：

- BLE 配网开始了
- 手机把 Wi‑Fi 密码发过来了
- Wi‑Fi 断开了
- 设备拿到 IP 了

这些都属于"事件"。

典型流程：

1. 你先启动 Wi‑Fi 或配网流程
2. 系统后台运行
3. 某一步完成后，触发事件
4. 事件回调函数被调用
5. 你在回调里更新状态、做下一步动作

## 2. 为什么 Wi‑Fi / 配网必须事件驱动

因为这些过程都需要时间，而且结果不确定。

所以系统设计成：

- **动作启动**
- **结果靠事件返回**

这就是你在 WiFiManager.c 里看到大量 event 和 handler 的原因。

## 3. 事件回调是什么

回调就是：

- 你先把一个处理函数交给系统
- 以后发生某类事件时，系统自动调用它

比如：

`esp_event_handler_register(WIFI_EVENT, ESP_EVENT_ANY_ID, &wifi_event_handler, NULL);`

意思就是：

- 以后只要有 WIFI_EVENT
- 就调用 wifi_event_handler()

所以"注册事件回调"本质上就是：

- 提前告诉系统："这类消息来了找我"

## 4. 在你这个项目里有哪些关键事件

你这个项目里最关键的是三类核心事件：

### NETWORK_PROV_EVENT

- 跟 BLE 配网流程有关
- 比如：
    - 配网开始
    - 收到 Wi‑Fi 凭据
    - 配网成功
    - 配网结束

### WIFI_EVENT

- 跟 Wi‑Fi 本身有关
- 比如：
    - Wi‑Fi station 启动
    - Wi‑Fi 断开

### IP_EVENT

- 跟 IP 获取有关
- 最关键的是：
    - IP_EVENT_STA_GOT_IP

## 5. 你应该建立的思维方式

以后看这种联网代码，要问：

- "我启动了什么流程？"
- "后续会收到哪些事件？"
- "每个事件会让状态跳到哪里？"

这就是嵌入式联网代码最重要的阅读方式。

# 8. DHCP / IP 基础

这一层是"设备真正联网成功"的关键。

## 1. 什么是 IP

IP 地址就是设备在网络里的地址。
你可以把它理解成：

- 设备在局域网里的门牌号

比如：

- 路由器是 192.168.1.1
- 设备可能拿到 192.168.1.103

这个 192.168.1.103 就是设备自己的 IP。

## 2. 为什么连上 Wi‑Fi 还不够

因为"连上 Wi‑Fi"只说明：

- 设备已经通过 SSID/密码接入了路由器

但还没说明：

- 路由器有没有给它分配网络地址
- 它能不能真正收发网络数据

所以真正的联网成功要再走一步：

- **获取 IP**

## 3. DHCP 是什么

DHCP 是自动分配 IP 的机制。

意思是：

- 设备连上路由器后
- 不需要你手动指定 IP
- 路由器会自动给它分配一个可用地址

比如：

- 路由器给设备分配 192.168.1.103

所以流程是：

1. 设备连上路由器
2. DHCP 开始工作
3. 路由器分配 IP
4. 设备收到 IP
5. 设备才真正具备网络通信能力

## 4. 你这个项目里哪个事件最关键

最关键的是：

- IP_EVENT_STA_GOT_IP

这表示：

- 设备已经拿到自己的 IP 地址了

在 WiFiManager 里，一旦收到这个事件：

- 状态就切到 CONNECTED

所以从工程角度你可以直接记住：

- WIFI_EVENT_STA_START 不是联网成功
- esp_wifi_connect() 不是联网成功
- IP_EVENT_STA_GOT_IP 才是联网成功的关键标志

## 5. 为什么 IP 这么重要

因为后面的所有网络请求都建立在它之上：

- HTTP 请求
- MQTT 连接
- RainMaker 通信
- 访问局域网服务

没有 IP：

- 设备没法成为网络里真正的参与者

# 9. 状态机思维

这是你以后做嵌入式联网和业务逻辑时必须掌握的思维。

一句话概括：

- **状态机就是把系统当前所处阶段抽象成几个稳定状态，再规定什么条件会让它切换。**

## 1. 为什么要用状态机

因为真实系统不是一直"只有一个 if else 就能讲清"。

比如联网流程里，设备可能处于：

- 空闲
- 配网中
- 正在连接
- 已连接
- 重连中
- 失败

如果不用状态机，代码很容易变成：

- 到处是标志位
- 到处是 if 嵌套
- 很难判断当前系统到底在什么阶段

状态机的好处是：

- 当前状态清楚
- 状态切换条件清楚
- 出问题容易定位

## 2. 你这个项目里的联网状态机

在 WiFiManager 里就是一个典型状态机：

- IDLE
- PROVISIONING
- CONNECTING
- CONNECTED
- RECONNECTING
- DISCONNECTED
- FAILED

每个状态都有明确含义。

比如：

- PROVISIONING：还在配网，用户还没把凭据给设备
- CONNECTING：已经有凭据，正在连路由器
- CONNECTED：拿到 IP 了
- RECONNECTING：之前连上过，但现在掉线重连

## 3. 状态机不是只有联网才有

你这个项目里还有另一个业务状态机：

- NORMAL
- REMIND
- ALARM
- SOS

那个在 AppController 里。

所以你要慢慢形成这个意识：

- 一个工程里往往不止一个状态机
- 联网是一个状态机
- 业务告警又是另一个状态机

## 4. 怎么看状态机

以后看状态机，不要一上来盯代码细节。
先问这三个问题：

1. 当前有哪些状态？
2. 每个状态是什么意思？
3. 什么事件会让它切换？

比如对 WiFiManager：

- 状态有哪些？
    - PROVISIONING / CONNECTING / CONNECTED ...
- 状态是什么意思？
    - 配网中 / 连网中 / 已联网
- 什么事件触发切换？
    - 配网开始、收到凭据、Wi‑Fi 启动、获取 IP、断线

## 5. 状态机思维真正重要的地方

不是背枚举值，而是学会这样思考：

- 系统现在在哪个阶段？
- 这时候允许做什么，不允许做什么？
- 下一个阶段是怎么触发的？
- 如果失败，应该退回哪个状态？

这在嵌入式里非常重要，因为设备是长期运行的，不是一次性脚本。

# 10. 安全意识

做 Wi‑Fi 配网时，安全意识非常重要，因为你处理的是：

- 用户家的 Wi‑Fi 名称
- 用户家的 Wi‑Fi 密码
- 设备的联网入口

如果这里做得太随意，后果可能是：

- 别人给设备乱配网
- 用户密码泄露
- 设备被接入错误网络
- 设备被恶意控制

## 1. 最核心的安全问题是什么

配网过程里最敏感的就是：

- **Wi‑Fi 凭据**

尤其是：

- password

因为一旦泄露，泄露的不是设备自己的秘密，而可能是：

- 用户整个家庭/办公网络的访问密码

所以你首先要建立这个意识：

- Wi‑Fi 密码属于高敏感信息

## 2. 什么叫"不能裸奔"

如果设备在配网时：

- 不做身份校验
- 不做安全会话
- 谁都能连上
- 谁都能发 Wi‑Fi 凭据

那就很危险。

这意味着：

- 附近任意人都可能给设备下发错误 Wi‑Fi
- 或者故意把设备配到恶意网络上

所以配网时至少要有一种"我确认对方是合法配网者"的机制。

在你这个项目里，这个角色就是：

- PoP

## 3. PoP 是什么，为什么重要

PoP = Proof of Possession

你可以把它理解成：

- 配网口令
- 配网时的身份凭证

作用是：

- 不是谁扫到这个 BLE 服务就能直接给设备发配置
- 还要知道正确的 PoP 才能建立安全会话

这能防止：

- 附近陌生人乱给设备配网

所以你要有这个安全意识：

- BLE 广播出来不代表任何人都应该能改配置
- 配网入口要有"最基本的门禁"

## 4. 为什么日志里打印密码是危险的

你项目里现在有一段日志会打印收到的 Wi‑Fi 密码。

这在调试时很方便，但安全上是有问题的。

因为：

- 串口日志可能被抓取
- 开发板演示时别人能看到
- 生产环境日志如果外泄，密码就泄露了

所以真实产品里应该做到：

- 可以打印 SSID
- 但不要明文打印 password
- 最多打印"已收到凭据"

这就是典型的：

- 开发方便性
- 和产品安全性
  之间的取舍

## 5. 要不要把配网信息长期留着

通常是要留着的，因为要支持重启自动联网。
但安全上你要知道：

- 存储凭据是必要的
- 但要清楚"设备里确实保存着敏感信息"

所以要有这些意识：

- 设备丢失时怎么办
- 恢复出厂时要不要清掉凭据
- 调试版和量产版处理方式可能不同

在你的项目里：

- ResetProvisioningAndRestart()
    就是"清掉敏感配网信息再重来"的入口

## 6. 配网服务暴露时间越短越好

从安全角度看，设备长期处于"可被配网状态"并不好。

因为这意味着：

- 攻击面长期暴露

更理想的策略通常是：

- 只有首次启动进入配网
- 或用户按键明确触发后才进入配网
- 配网结束后关闭 BLE 配网服务

你的项目已经有一点这个思路：

- 清除凭据后才重新进入配网

这就是比较合理的产品思维。

## 7. 你作为嵌入式工程师至少该有的安全意识

在 Wi‑Fi 配网这个流程上，你至少应该能主动想到这些问题：

- 有没有把密码明文打印出来
- 配网是不是谁都能发起
- 配网结束后有没有及时关闭入口
- 恢复出厂会不会清掉旧凭据
- 手机和设备之间有没有最基本的身份校验
- 设备是否会被误配到错误网络

你不一定一开始就把所有安全都做满，但一定要先有"哪里危险"的意识。

# 11. 故障排查能力

这是实际工程里最有价值的能力之一。

因为配网问题非常容易卡在不同层，你必须学会判断：

- 到底卡在哪一步了

## 1. 先建立"分层排查"思维

配网流程不是一个点，而是一整条链。
排查时不要笼统问：

- "为什么设备联网失败了？"

而要拆成阶段：

1. 设备有没有进入配网状态
2. 手机有没有找到 BLE 服务
3. 手机有没有成功连上 BLE
4. 凭据有没有发到设备
5. 设备有没有保存凭据
6. 设备有没有开始连 Wi‑Fi
7. Wi‑Fi 有没有连上路由器
8. 有没有拿到 IP
9. 拿到 IP 后能不能访问服务器

这样你才能快速定位问题属于哪一层。

## 2. 常见故障点有哪些

你至少要会区分这几类：

### 第一类：BLE 侧就没起来

表现可能是：

- 手机扫不到设备
- 日志里没有 BLE provisioning started

说明问题可能在：

- 配网服务没启动
- BLE 初始化失败
- 状态根本没进入 PROVISIONING

### 第二类：手机连上了 BLE，但没发成功凭据

表现可能是：

- 能看到设备
- 但没有 NETWORK_PROV_WIFI_CRED_RECV

说明问题可能在：

- 手机端协议不匹配
- PoP 错误
- 安全会话没建立成功

### 第三类：收到凭据了，但 Wi‑Fi 连不上

表现可能是：

- 有 received Wi‑Fi credentials
- 但后面一直连不上
- 或频繁 STA_DISCONNECTED

说明问题可能是：

- 密码错误
- 路由器不存在
- 信号差
- 认证失败

### 第四类：Wi‑Fi 连上了，但没拿到 IP

表现可能是：

- 看到连接动作了
- 但一直没有 IP_EVENT_STA_GOT_IP

说明问题可能在：

- DHCP 异常
- 路由器分配失败
- 网络环境问题

### 第五类：拿到 IP 了，但业务还是不通

表现可能是：

- wifi connected, ip=...
- 但 HTTP / MQTT 发不出去

说明问题不是"配网失败"，而是：

- 服务器不可达
- DNS 问题
- 上层应用问题
- 云平台认证问题

这一点特别重要：

- **拿到 IP 说明联网成功**
- **但不代表业务链一定正常**

## 3. 怎么排查最快

最实用的方法是看"关键日志节点"。

你这个项目最关键的检查点可以按顺序看：

1. Wi-Fi not provisioned, starting BLE provisioning
2. BLE provisioning started
3. BLE transport connected
4. received Wi-Fi credentials ssid=...
5. provisioning successful
6. wifi station started
7. wifi connected, ip=...

你可以把它当成一条流水线。

日志断在哪一步，问题通常就在哪一层。

## 4. 为什么要会看状态，而不只看错误码

错误码当然重要，但有时最有价值的是：

- 当前状态是什么
- 状态是怎么走到这里的

比如：

- 一直停在 PROVISIONING
    - 说明手机侧还没把凭据走通
- 停在 CONNECTING
    - 说明凭据可能有了，但 Wi‑Fi 还没成
- 进过 CONNECTED 又掉回 RECONNECTING
    - 说明不是配网问题，是联网稳定性问题

这就是状态机思维和故障排查能力结合起来的地方。

## 5. 你作为嵌入式工程师最该养成的排查习惯

我建议你一定要养成下面这个习惯：

遇到联网问题，先问自己：

1. 设备现在处于什么状态？
2. 上一个成功事件是什么？
3. 下一个本该出现的事件为什么没来？
4. 是 BLE 问题、Wi‑Fi 问题、IP 问题，还是上层业务问题？

这四个问题比"盲猜代码哪里错了"有用得多。

## 6. 对你当前项目最实用的排查能力

结合你现在这个工程，最值得练的是：

- 会根据日志判断卡在配网链的哪一段
- 会区分：
    - 配网没完成
    - Wi‑Fi 没连上
    - IP 没拿到
    - 云端上报失败
- 会通过长按 8 秒清掉配网信息，重新走完整流程验证

这就已经是很实用的工程能力了。

# 12. 对你这个项目，最该掌握的具体点

如果完全贴着你现在的代码，我建议你一定要掌握这 12 个点：

1. **WiFiManager_Init() 做了哪些初始化**

   按代码顺序，一共做了 **6 层初始化 + 1 次判断 + 1 次分支决策**：
   - **NVS 初始化** (`init_nvs_once`) — 调用 `nvs_flash_init()`，为后续读/写配网凭据做准备
   - **网络接口框架** (`init_netif_once`) — 调用 `esp_netif_init()`，为创建网络接口打底
   - **默认事件循环** (`init_event_loop_once`) — 调用 `esp_event_loop_create_default()`，让系统能收发异步事件
   - **创建 STA netif** — `esp_netif_create_default_wifi_sta()`，建立设备的"无线网卡对象"
   - **初始化 Wi-Fi 驱动** — `esp_wifi_init(&cfg)`，其中 `cfg` 是 `wifi_init_config_t` 类型变量，由 `WIFI_INIT_CONFIG_DEFAULT()` 宏生成默认配置参数（TX/RX buffer 大小、NVS 开关等）
   - **注册 5 类事件回调** — 把 `wifi_event_handler` 注册到 NETWORK_PROV_EVENT、WIFI_EVENT、IP_EVENT、PROTOCOMM_TRANSPORT_BLE_EVENT、PROTOCOMM_SECURITY_SESSION_EVENT
   - **判断是否已配网** — `network_prov_mgr_is_wifi_provisioned()`
   - **分支**：已配网 → `start_wifi_sta()` 直接连；未配网 → `start_ble_provisioning()` 走 BLE 配网

2. **network_prov_mgr_is_wifi_provisioned() 为什么是分叉点**

   因为它是 **整个联网流程的分水岭**（代码 427-434 行）：
   - 如果返回 `true` → 设备**以前配过网**，NVS 里有凭据 → 只需要切到 STA 模式去连路由器
   - 如果返回 `false` → 设备**是第一次上电或配网信息被清过** → 必须走 BLE 配网让用户把凭据送进来
   
   后面的所有逻辑（连 Wi-Fi 还是开 BLE）都取决于这个判断。

3. **start_ble_provisioning() 是怎么启动配网的**

   流程如下（代码 293-320 行）：
   1. **生成服务名** — 调用 `get_device_service_name()`，用 MAC 后三字节拼出 `PROV_xxxxxx` 格式的名字
   2. **设置自定义 UUID** — 调用 `network_prov_scheme_ble_set_service_uuid()`，告诉 BLE 层用哪个 UUID 广播配网服务
   3. **标记配网状态** — `s_provisioning_active = true`，状态切到 `PROVISIONING`
   4. **真正启动配网** — 调用 `network_prov_mgr_start_provisioning(NETWORK_PROV_SECURITY_1, POP, service_name, NULL)`，这步让系统开始 BLE 广播并等待手机连接
   5. **打印二维码信息** — `log_provisioning_qr()` 把服务名、PoP 等信息拼成 JSON 打印出来，供手机端 ESP RainMaker 扫码配网
   
   如果启动失败，会回退清理（`deinit`）并把状态设为 `FAILED`。

4. **service_name、PoP、UUID 分别干嘛**

| 参数               | 值举例             | 作用                                                  |
| ---------------- | --------------- | --------------------------------------------------- |
| **service_name** | `PROV_A1B2C3`   | 给人看的设备名字。手机扫描 BLE 时看到的设备名，方便用户从列表中识别出自己那台设备         |
| **PoP**          | `eldercare1234` | 配网口令 / 身份凭证。防止附近无关人员给设备乱配网，手机必须知道正确的 PoP 才能建立安全会话   |
| **UUID**         | `b4df5a1c-...`  | 给程序看的服务标识。手机通过这个 UUID 判断设备的 BLE 服务是不是配网服务，而不是其他蓝牙设备 |
   
   代码中：service_name 由 `WIFI_MANAGER_PROV_SERVICE_PREFIX` + MAC 生成；PoP 由宏 `WIFI_MANAGER_PROV_POP` 定义；UUID 在 `start_ble_provisioning()` 里通过 `network_prov_scheme_ble_set_service_uuid()` 设置。

5. **wifi_event_handler() 为什么是核心**

   因为它承担了 **统一的异步事件入口** 的角色（代码 135-269 行）。
   
   WiFiManager 的状态切换完全是事件驱动的，这个回调函数统一接收 **5 类事件**：
   - `NETWORK_PROV_EVENT` — 配网流程
   - `WIFI_EVENT` — Wi-Fi 启停、断开
   - `IP_EVENT` — IP 获取
   - `PROTOCOMM_TRANSPORT_BLE_EVENT` — BLE 传输层连接/断开
   - `PROTOCOMM_SECURITY_SESSION_EVENT` — 安全会话建立/失败
   
   没有这个回调，系统就收不到任何异步通知，状态机也无法推进。它把所有事件汇集到一个地方处理，避免了分散的回调管理。

6. **NETWORK_PROV_EVENT 处理什么**

   处理 BLE 配网过程中的 5 个事件（代码 142-189 行）：

   | 事件 | 作用 |
   |------|------|
   | `NETWORK_PROV_START` | 标记配网中，切到 `PROVISIONING` |
   | `NETWORK_PROV_WIFI_CRED_RECV` | 收到手机发来的 SSID/密码 |
   | `NETWORK_PROV_WIFI_CRED_FAIL` | 凭据无效，退回 `PROVISIONING` |
   | `NETWORK_PROV_WIFI_CRED_SUCCESS` | 凭据有效，切到 `CONNECTING` |
   | `NETWORK_PROV_END` | 配网结束，清理资源 |


7. **WIFI_EVENT 处理什么**

   处理 Wi-Fi 驱动层面的 2 个关键事件（代码 192-221 行）：

   | 事件 | 作用 |
   |------|------|
   | `WIFI_EVENT_STA_START` | Wi-Fi 已启动，调用 `esp_wifi_connect()` 去连路由器 |
   | `WIFI_EVENT_STA_DISCONNECTED` | 断线，清 IP，自动重连 |

   注意：`WIFI_EVENT_STA_START` 不等于连上网，它只是说"无线功能打开了，现在开始去拨号"。

8. **IP_EVENT_STA_GOT_IP 为什么才算真正联网成功**

   因为前面的步骤都不代表设备能正常收发网络数据。看代码（224-238 行）：
   
   收到此事件时：
   1. 把 IP 地址保存到 `s_ip_string`
   2. 状态切到 `CONNECTED`
   3. 打印 `wifi connected, ip=...`
   
   对照 md 第 8 节的说法：
   - **Wi-Fi 连上** = 通过了路由器的认证，但还没拿到门牌号
   - **DHCP 分配 IP** = 路由器给了设备一个局域网地址
   - **拿到 IP** = 设备才有在网络里收发数据的能力
   
   所以 `WIFI_EVENT_STA_START` 和 `esp_wifi_connect()` 都不算联网成功，`IP_EVENT_STA_GOT_IP` 才是。代码里 `WiFiManager_IsConnected()` 也是判断状态是否等于 `CONNECTED`，而这个状态只有收到 `IP_EVENT_STA_GOT_IP` 时才会被设置。

9. **配网信息为什么能跨重启保存**

   两段代码协作完成：
   1. **WiFiManager_Init()** 一开始就做了 `nvs_flash_init()`（代码 56-67 行），确保 NVS 子系统可用
   2. **配网协议层** 把手机发来的 SSID/密码写进了 NVS 分区（这是 `network_prov_mgr` 内部做的，WiFiManager 不直接操作）
   
   NVS 是 **非易失性存储**，断电后数据不丢失。所以下次启动时：
   - `nvs_flash_init()` 重新打开 NVS
   - `network_prov_mgr_is_wifi_provisioned()` 读到之前保存的凭据标记 → 返回 `true`
   - 不走 BLE 配网，直接启动 STA 去连 Wi-Fi
   
   这样用户只需要配一次网，设备断电重启后会自动连上。

10. **WiFiManager_IsConnected() 为什么能被上层用**

    因为它的实现是一个 **纯查询函数**（代码 468-471 行）：
    
    ```c
    bool WiFiManager_IsConnected(void)
    {
        return s_state == WIFI_MANAGER_STATE_CONNECTED;
    }
    ```
    
    它只做一件事：**返回当前状态是不是 `CONNECTED`**。而上层（比如 `AppController`）不需要知道 Wi-Fi 驱动细节、不需要监听事件、不需要关心配网还是重连，只需要问一句"网通了吗"就行了。
    
    这种设计的好处是：
    - **解耦** — AppController 不关心联网细节，只需要知道结果
    - **状态集中管理** — 联网状态只有 `wifi_event_handler` 能改，不会出现多处乱改的问题
    - **接口稳定** — 底层实现怎么改，只要 `WiFiManager_IsConnected()` 语义不变，上层不用动

11. **长按 8 秒为什么会重新进入配网**

    核心在 `WiFiManager_ResetProvisioningAndRestart()`（代码 449-461 行）：
    
    ```c
    esp_err_t WiFiManager_ResetProvisioningAndRestart(void)
    {
        network_prov_mgr_reset_wifi_provisioning();  // ① 清 NVS 里的凭据
        vTaskDelay(pdMS_TO_TICKS(200));               // ② 等 200ms 确保写完成
        esp_restart();                                 // ③ 重启设备
    }
    ```
    
    三步走：
    1. **清凭据** — `network_prov_mgr_reset_wifi_provisioning()` 删掉 NVS 里保存的 Wi-Fi SSID/密码
    2. **等写入完成** — 200ms 延迟保证 flash 写操作完成
    3. **重启** — 设备重新跑 `WiFiManager_Init()`，这时 `network_prov_mgr_is_wifi_provisioned()` 返回 `false`，就会走 `start_ble_provisioning()` 重新进入配网流程
    
    所以"长按 8 秒"这个动作映射到代码就是：**清凭据 → 重启 → 重新进入 BLE 配网**。

12. **RainMakerReporter 和配网不是一回事**
    RainMakerReporter 负责配网成功后向云端上报数据，它依赖配网拿到的 IP 连接云平台。配网是"让设备能上网"，Reporter 是"上网后做的事"。
