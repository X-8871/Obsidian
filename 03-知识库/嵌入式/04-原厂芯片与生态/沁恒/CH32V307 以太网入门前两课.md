## 这份文档是干什么的

  

这份笔记是给当前项目做以太网上云前的入门资料。

  

目标只有两个：

  

1. 先把网络里最常见的专有名词讲清楚。

2. 先把整个项目的数据流和通信框架讲清楚。

  

等你把这两部分看明白，后面我们再一起改代码、调网口、做上云，你就不会只是在照抄。

  

---

  

## 第一课：先认识这些专有名词

  

### 1. 整个网络系统可以分成 4 层

  

你可以把网络理解成 4 层分工：

  

1. 硬件层

2. 网络层

3. 传输层

4. 应用层

  

对应到你这个项目里：

  

- 硬件层：网口、网线、PHY、MAC

- 网络层：IP 地址、网关、子网掩码

- 传输层：UDP、TCP、端口

- 应用层：HTTP、MQTT、JSON

  

以后看到网络项目，先问自己四个问题：

  

1. 硬件通了吗？

2. IP 对了吗？

3. 走 UDP 还是 TCP？

4. 上层说的是 HTTP 还是 MQTT？

  

---

  

### 2. MAC 和 PHY 是谁

  

这是做以太网最先会碰到的两个词。

  

#### MAC

  

MAC 是单片机内部负责以太网数据帧收发控制的模块。

  

你可以把它理解成：

  

`负责按以太网规则打包、拆包、收发数据的控制模块`

  

它主要负责：

  

- 组织以太网帧

- 控制发送和接收流程

- 和协议栈配合工作

  

CH32V307 自带 MAC。

  

#### PHY

  

PHY 是物理层模块，负责把数字信号真正变成网线上的电信号。

  

你可以把它理解成：

  

`负责让电信号真的跑上网线`

  

它主要负责：

  

- 检测网线有没有插好

- 检测链路是否建立

- 完成物理信号收发

  

CH32V307 的一个重要特点是：它内部带 10M PHY，所以做 10M 以太网时不一定要额外外接 PHY 芯片。

  

#### 一句话记忆

  

- MAC：负责“按规则收发”

- PHY：负责“真正上网线”

  

---

  

### 3. 什么是 Link Up / Link Down

  

这表示物理链路是否建立。

  

- `Link Up`：网线、电路、PHY 基本正常，物理层通了

- `Link Down`：物理层还没通，后面 IP、UDP、MQTT 都不用谈

  

所以以后排查网络问题，第一步永远先看：

  

`有没有 link up`

  

如果没有 `link up`，大概率先查这些：

  

- 网线是否正常

- 水晶头和变压器是否接对

- 硬件时钟是否正确

- PHY 引脚或硬件设计是否有问题

  

---

  

### 4. MAC 地址和 IP 地址是什么

  

这两个概念很像，但不是一回事。

  

#### MAC 地址

  

MAC 地址更像设备的底层硬件身份证。

  

特点：

  

- 一般和硬件相关

- 在局域网底层通信中使用

- 格式通常像 `00-11-22-33-44-55`

  

#### IP 地址

  

IP 地址更像设备在网络里的门牌号。

  

特点：

  

- 通信时更常用

- 表示设备在当前网络中的地址

- 例如 `192.168.137.10`

  

#### 一句话记忆

  

- MAC 地址：设备底层身份

- IP 地址：设备当前网络位置

  

---

  

### 5. 什么叫同网段

  

这对调试特别重要。

  

例如：

  

- 单片机 IP：`192.168.137.10`

- 电脑 IP：`192.168.137.1`

- 子网掩码：`255.255.255.0`

  

这通常说明它们在同一个局域网里，可以直接互相通信。

  

如果变成：

  

- 单片机：`192.168.137.10`

- 电脑：`192.168.1.100`

  

那通常就不在同一个网段，直接通信往往不行。

  

所以以后说“先检查 IP”，核心就是检查：

  

- IP 地址是否合理

- 子网掩码是否一致

- 双方是否同网段

  

---

  

### 6. TCP 和 UDP 是什么

  

它们都属于传输层协议，负责“怎么送数据”。

  

#### UDP

  

特点：

  

- 简单

- 快

- 开销小

- 不保证对方一定收到

  

适合：

  

- 局域网调试

- 传感器周期上报

- 实时性要求高、允许偶尔丢包的场景

  

你当前工程里就是 UDP。

  

#### TCP

  

特点：

  

- 更可靠

- 需要先建立连接

- 会确认数据有没有收到

- 相对更复杂

  

适合：

  

- 重要数据传输

- HTTP

- MQTT 的底层连接

  

#### 一句话记忆

  

- UDP：轻量、快、可能丢包

- TCP：可靠、稳定、但更复杂

  

---

  

### 7. 端口是什么

  

IP 地址像一栋楼的门牌号，端口像这栋楼里的房间号。

  

例如：

  

- IP：`192.168.137.1`

- 端口：`5005`

  

意思就是：

  

`把数据发给 192.168.137.1 这台设备上的 5005 号程序`

  

所以通信时往往同时要知道两件事：

  

- 发给哪台设备：IP

- 发给设备上的哪个程序：端口

  

---

  

### 8. 协议栈是什么

  

协议栈就是一套已经写好的网络规则代码。

  

因为你不可能自己从零实现这些内容：

  

- ARP

- IP

- ICMP

- UDP

- TCP

- DHCP

- DNS

  

所以厂商会给你一套网络库。你现在项目里用的是 `WCHNET`。

  

你可以把协议栈理解成一个网络发动机：

  

- 初始化：相当于点火

- 周期驱动：相当于让发动机持续运转

  

你工程里已经有这两个关键动作：

  

- `WCHNET_TimeIsr(WCHNETTIMERPERIOD)`

- `WCHNET_MainTask()`

  

这说明协议栈不是初始化完就自己永远跑下去，而是需要你不断“喂时间片”和“驱动主任务”。

  

---

  

### 9. HTTP 是什么

  

HTTP 是网页和服务器最常见的应用层协议。

  

你打开网页，本质上就在用 HTTP。

  

特点：

  

- 很常见

- 适合请求-响应模式

- 常用于网页和云接口

  

例如单片机或者电脑向服务器发一条 HTTP 请求：

  

```json

{

  "temp": 25.6,

  "hum": 60.2

}

```

  

这类数据很常见会作为 HTTP 的内容上传。

  

HTTP 适合：

  

- 调用 Web API

- 上传数据到普通服务器

  

缺点是：

  

- 对单片机来说比 UDP 更重

- 如果再加 HTTPS/TLS，会更复杂

  

---

  

### 10. MQTT 是什么

  

MQTT 是物联网里很常见的应用层协议。

  

它和 HTTP 最大的区别是：

  

- HTTP 更像“我主动去请求一次”

- MQTT 更像“我接入消息服务器，持续发布和订阅消息”

  

MQTT 里有几个重要角色：

  

- `Client`：客户端，比如设备、电脑程序、手机 App

- `Broker`：消息服务器，中转消息

- `Topic`：主题，像频道名

  

比如：

  

- 你的设备往某个 topic 发布温度数据

- 云平台订阅这个 topic

- 手机或网页也可以订阅这个 topic

  

MQTT 特别适合物联网，因为它适合：

  

- 周期上传

- 设备在线消息交互

- 多端同时接收同一份数据

  

---

  

### 11. HTTP 和 MQTT 的区别

  

可以先这样记：

  

#### HTTP

  

- 更像“请求一次，回应一次”

- 常用于网页和接口

  

#### MQTT

  

- 更像“持续接入消息系统”

- 常用于物联网平台

  

如果你是做传感器上云：

  

- 想先简单调通局域网：先 UDP

- 想接入物联网平台：常见选择是 MQTT

- 想对接普通 Web 接口：常见选择是 HTTP

  

---

  

### 12. JSON 是什么

  

JSON 不是网络协议，它只是数据格式。

  

例如：

  

```json

{

  "temp_x10": 256,

  "hum_x10": 603,

  "alarm": 0

}

```

  

这只是“数据长什么样”，不是“怎么传”。

  

同样一份 JSON 可以通过很多方式传：

  

- UDP 发送

- TCP 发送

- HTTP body 发送

- MQTT payload 发送

  

所以一定要分清：

  

- UDP / TCP / HTTP / MQTT：是传输或应用协议

- JSON：是数据内容格式

  

---

  

### 13. DHCP、DNS、TLS 是什么

  

#### DHCP

  

自动分配 IP 地址的协议。

  

如果开启 DHCP，设备上电后会自动申请 IP，不用你手动写死。

  

#### DNS

  

把域名解析成 IP 地址。

  

例如把：

  

`iot.example.com`

  

解析成真正的服务器 IP。

  

#### TLS

  

加密通信。

  

例如：

  

- `HTTPS = HTTP + TLS`

- `MQTTS = MQTT + TLS`

  

TLS 的作用是：

  

- 防止数据被偷看

- 防止数据被篡改

  

但对单片机来说，TLS 往往比较难，因为它更占资源、配置也更复杂。

## 第二课：把整个项目通信框架讲清楚

  

这一课的目标不是继续背名词，而是搞明白：

  

`谁在和谁说话，谁负责哪一段工作`

  

你当前项目最推荐、也最稳的路线是：

  

```text

传感器

  ↓

CH32V307 单片机

  ↓

WCHNET 协议栈

  ↓

UDP 数据包

  ↓

网线 / 路由器 / 电脑网卡

  ↓

电脑上的 Python 桥接程序

  ↓

1. 本地网页看板

2. 阿里云 MQTT

```

  

---

  

### 1. 传感器层在做什么

  

传感器的职责很单纯：

  

`负责产生原始数据`

  

例如：

  

- 温度

- 湿度

- 气压

- 光照

- 烟雾

  

传感器本身不会上网，它只是把数据交给单片机。

  

---

  

### 2. CH32V307 在做什么

  

CH32V307 是整个系统的大脑。

  

它负责：

  

- 读取传感器

- 整理数据

- 打包数据

- 交给网络模块发送

  

例如，单片机会把采集到的数据整理成一段 JSON：

  

```json

{

  "temp_x10": 256,

  "hum_x10": 603,

  "lux_x10": 1280

}

```

  

所以 CH32V307 的职责可以概括为：

  

`把传感器数据变成可发送的数据`

  

---

  

### 3. WCHNET 在做什么

  

WCHNET 是网络协议栈，是你的网络总管。

  

你不需要自己从零实现：

  

- IP 封装

- UDP 封装

- ARP 查询

- 以太网数据帧收发

  

这些都由 WCHNET 负责。

  

流程可以理解为：

  

```text

你的应用代码

  ↓

WCHNET

  ↓

MAC/PHY

  ↓

网线

```

  

所以 WCHNET 的职责是：

  

`把“我要发数据”变成真正的网络数据包`

  

---

  

### 4. UDP socket 在做什么

  

`socket` 可以理解成一个网络通信窗口。

  

如果你想发数据，不是直接“喊一声发出去”，而是先创建一个通信口。

  

你当前工程里用的是 `UDP socket`，它负责：

  

- 创建通信通道

- 指定目标 IP

- 指定目标端口

- 周期性发送数据

  

你可以把它想成：

  

`我开了一个寄快递的窗口，专门往某个地址寄数据包`

  

---

  

### 5. 网线和局域网在做什么

  

这一层是传输通道。

  

数据从单片机出来后，会经过：

  

- PHY

- 网线

- 路由器或交换机

- 电脑网卡

  

这层最关心两件事：

  

- 物理连接是否正常

- IP 是否可达

  

它的职责是：

  

`把数据从单片机送到电脑`

  

---

  

### 6. 电脑桥接程序在做什么

  

这是你当前项目里非常重要的角色。

  

电脑上的 Python 桥接程序相当于：

  

`翻译器 + 中转站`

  

它主要做两件事：

  

1. 接收 MCU 发来的 UDP 数据

2. 把这份数据转发给别的系统

  

它可以转发到：

  

- 本地网页看板

- 阿里云 MQTT

  

流程像这样：

  

```text

CH32V307 -> UDP -> Python桥接程序 -> MQTT -> 阿里云

```

  

它为什么重要？

  

因为很多复杂问题可以先交给电脑处理：

  

- MQTT 连接

- TLS 加密

- 云平台认证

- 更详细的调试日志

  

所以桥接程序的职责是：

  

`把单片机容易发出的数据，变成云平台容易接收的数据`

  

---

  

### 7. 本地网页看板在做什么

  

本地网页看板不仅是展示工具，更是调试工具。

  

它可以帮助你确认：

  

- 电脑是否收到 MCU 的 UDP 数据

- 数据格式是否正确

- 桥接程序是否工作正常

  

这在排错时特别重要。

  

例如：

  

- 如果本地网页有数据，但云端没有数据

- 那就说明 MCU 到电脑这段大概率是通的

- 问题更可能出在 MQTT 配置或云平台参数

  

---

  

### 8. 阿里云 MQTT 在做什么

  

阿里云这一层可以理解成“云端消息入口”。

  

电脑桥接程序会作为 MQTT 客户端连接阿里云，然后把接收到的传感器数据发布到指定 topic。

  

流程可以概括为：

  

```text

CH32V307 采集数据

  ↓

发 UDP 给电脑

  ↓

Python 程序收到 JSON

  ↓

封装为 MQTT 消息

  ↓

发布到阿里云 topic

```

  

然后阿里云可以继续做：

  

- 存储数据

- 规则引擎处理

- 远程展示

- 和其他系统联动

  

---

  

### 9. 整个项目其实是“三段通信”

  

你这个系统不是一种协议从头走到尾，而是三段式：

  

```text

传感器协议 -> 单片机本地处理 -> UDP -> 电脑桥接 -> MQTT -> 云端

```

  

更具体地说：

  

#### 第一段：传感器和单片机通信

  

这一段通常用：

  

- I2C

- ADC

- GPIO

  

这还不属于网络通信。

  

#### 第二段：单片机和电脑通信

  

这一段用：

  

- UDP

  

这是局域网通信。

  

#### 第三段：电脑和云平台通信

  

这一段常用：

  

- MQTT

  

这是上云通信。

  

---

  

### 10. 为什么不建议一开始就让 MCU 直接连公网云

  

对于新手来说，直接让 CH32V307 连公网云平台，难点通常集中在：

  

- MQTT 参数更多

- 云平台认证更复杂

- TLS 难度较高

- 调试不方便

  

而拆成三段之后：

  

- 单片机只负责局域网 UDP，简单

- 电脑负责 MQTT/TLS，灵活

- 云平台只负责收数据

  

这种方式成功率更高，也更适合比赛、课程设计和入门项目。

  

---

  

### 11. 以后调试项目时的标准检查顺序

  

以后我们一起做这个项目时，你就按下面顺序查问题：

  

1. 传感器读到数据了吗？

2. 网口 `link up` 了吗？

3. 单片机有没有发 UDP？

4. 电脑有没有收到 UDP？

5. 电脑有没有成功连接阿里云 MQTT？

6. 云端有没有显示数据？

  

只要按这个顺序排查，问题就不会乱。

  

---

  

## 第二课小结

  

这节课最重要的不是背新词，而是建立这个脑图：

  

```text

传感器负责出数据

CH32V307 负责采集和发送

WCHNET 负责按网络协议打包和发包

UDP 负责把数据送到电脑

电脑桥接程序负责转成 MQTT

阿里云负责接收和管理数据

```

  

记住一句核心话：

  

`CH32V307 不是直接“把数据上传到云”，而是先把数据发到网络上，再由系统中的下一段继续处理。`

  

---

  

## 最后的复习建议

  

如果你暂时还是会混淆这些名词，建议你先反复记下面这几组关系：

  

- MAC 不是 PHY

- IP 不是端口

- UDP/TCP 不是 JSON

- HTTP/MQTT 是应用层协议

- JSON 是数据格式

- WCHNET 是协议栈

- 桥接程序不是单片机代码，而是电脑侧程序

  

只要这些关系你理顺了，后面进入代码实现阶段就会顺很多。
