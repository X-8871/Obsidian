# 模块 5：自动微分 (Autograd)

在上一章中，我们千辛万苦搭好了一个神经网络大厨。
但是，刚招来的大厨是个**纯小白**！他脑子里的菜谱（也就是神经网络内部成千上万个 `weight` 和 `bias` 参数）全都是随机瞎写的。如果让他做这道菜，他做出来的味道（预测结果）一定会极其难吃。

**问题来了：大厨怎么才能进步？**
答案是：**品菜师（Loss Function）**尝一口，大骂一句“太咸了！”，然后大厨必须知道这口锅里哪几种调料放多了，下一次少放点。这个“寻找是哪种调料导致了难吃”的复盘反思过程，在深度学习中叫做**反向传播 (Backpropagation)**。

在很多年前，手写反向传播（求导数）是博士生才会干的苦力活。
但今天，PyTorch 提供了一个叫做 **Autograd（自动微分引擎）** 的超级黑科技。它能在后台偷偷记录大厨的每一个动作，并在品菜师发火时，瞬间算出每一项参数该怎么微调！

---

## 1. 开启“全天候监控”：`requires_grad=True`

要想让 Autograd 工作，大厨身上的每一个参数（权重 $w$ 和偏置 $b$）必须带有一个特殊的监控装置：`requires_grad=True`（需要计算梯度）。
*幸运的是，我们在上一章用的 `nn.Linear` 里面的所有参数，PyTorch 已经在底层默认帮我们开启了这个监控。*

为了看清底层逻辑，我们手动写一个极简的计算过程：
```python
import torch

# 输入数据 x，正确答案 y
x = torch.ones(5)  # input tensor
y = torch.zeros(3)  # expected output

# 大厨脑子里的两个参数 w 和 b。我们手动给它们带上监控装置！
w = torch.randn(5, 3, requires_grad=True)
b = torch.randn(3, requires_grad=True)

# 大厨开始做菜 (前向传播)
z = torch.matmul(x, w) + b

# 品菜师打分：计算难吃程度 (这里用一个极简的计算代表 Loss)
# 我们用 F.binary_cross_entropy_with_logits 来算分
loss = torch.nn.functional.binary_cross_entropy_with_logits(z, y)
```

在这个过程中，Autograd 就像一个密探，它默默地画出了一张**计算图 (Computational Graph)**。它记录了：`w` 和 `x` 做了乘法，然后加了 `b` 变成了 `z`，最后 `z` 变成了 `loss`。

---

## 2. 品菜师发火：`.backward()` 反向传播

当算出了 `loss`（难吃程度）之后，我们要让大厨去反思。
只需要简单粗暴地执行一句话，魔法就会发生：

```python
# 也就是大吼一声：给我沿着计算图往回找原因！
loss.backward()
```

就这一瞬间，Autograd 引擎沿着它刚才画的那张图，**一路从后往前倒推（利用高等数学里的链式求导法则）**，算出了“这口锅这么难吃，`w` 和 `b` 各自要负多大的责任”。

这个责任值（在数学上叫作**梯度 Gradient**），被悄悄地存进了参数自己的口袋里：
```python
print(f"w 应该负的责任 (w.grad): \n{w.grad}\n")
print(f"b 应该负的责任 (b.grad): \n{b.grad}\n")
```
在下一章，我们的大厨就会根据这些存起来的责任值（梯度），去纠正自己的菜谱。

---

## 3. 关掉监控探头：`torch.no_grad()`

Autograd 的监控探头非常耗费内存和计算力。
在**训练**大厨的时候，我们必须开着探头让他反思。
但是，当大厨出师了，真的去餐厅给客人做菜时（在这个阶段叫作 **模型评估 / 测试 / 推理阶段**），客人是不会去纠正大厨的！这时候如果还开着探头，纯粹是浪费电。

所以，我们要学会一个关掉探头的咒语：**`torch.no_grad()`**

```python
# 假设现在是正式营业时间
# 使用 with torch.no_grad(): 把代码包裹起来
with torch.no_grad():
    # 在这个区域里做的所有前向计算，后台都不会画计算图，也不会监控
    z = torch.matmul(x, w) + b
print(z.requires_grad) # 输出必定是 False！
```

这不仅能**加快运行速度**，还能**大幅度省下显卡内存**！

---

## 总结
1. **`requires_grad=True`**：给参数挂上监控探头，允许 Autograd 后台画图记录。
2. **`loss.backward()`**：反向传播的核按钮！一键算出所有监控参数的梯度（责任值）。
3. **`w.grad`**：计算完后，参数该怎么改的“指示纸条”就塞在这个口袋里。
4. **`torch.no_grad()`**：当不需要模型再学习，只要它干活时，用这个咒语关掉监控，省电省内存。

## 4. Q&A 补充精华（基于你的深度提问）

在互动中，我们理清了 Autograd 和深度学习底层最关键的几个疑惑：

### 4.1 权重为什么会有负数？
如果把权重仅仅理解为“重要程度”，很难理解负数。在神经网络里，权重代表的是**“影响的方向”**：
*   **正权重**：加分项（赞成票）。比如看鞋的时候发现鞋跟，疯狂加分。
*   **负权重**：扣分项（一票否决）。比如看鞋的时候发现一堆厚棉布，立刻扣除极大的分数，排斥它是高跟鞋的可能。

### 4.2 `torch.randn` 里的 `n` 是什么？
它是 **Normal (正态分布/高斯分布)**。普通的 `rand` 只能生成 0~1 的随机正数。而带了 `n` 的 `randn` 会以 0 为中心，生成有正有负的小数。对于大厨脑子的初始化来说，这种带有正负偏好的正态分布能让网络学得更快更稳。

### 4.3 为什么 Loss 函数的名字那么长？
`binary_cross_entropy_with_logits` 拆开来看其实是大白话：
*   `binary`：做二选一任务的。
*   `cross_entropy`：计算误差的经典数学公式（交叉熵）。
*   `with_logits`：自带对生肉（未经转换的原始打分 logits）的解析能力，不用你提前手算 Softmax。
*(程序员偷懒绝招：实战中通常会使用它的缩写类 `nn.BCEWithLogitsLoss()`)*

### 4.4 `w.grad` 到底是什么时候被填满数值的？
**时间轴极其精确**：
1. 造出 `w` 时：`w.grad` 是 `None`。
2. 前向传播（炒菜）做完时：`w.grad` 是 `None`（探头只画图不追责）。
3. 算出 `loss`（品菜师打分）时：`w.grad` 依然是 `None`。
4. **按下核按钮 `loss.backward()` 的瞬间**：Autograd 顺着计算图往回倒推，瞬间把算好的责任值（梯度）塞进 `w.grad` 这个口袋里！

### 4.5 什么时候关探头 (`torch.no_grad()`)？
**训练集（后厨培训期）**：需要反向传播来学习，所以必须开着探头。
**测试集（正式营业期）**：大厨只需要做正向传播（看图做菜）给出结果，客人们不会去纠正大厨的做法。所以完全不需要反向传播，此时必须用 `torch.no_grad()` 关掉探头，省显存、提速度！
